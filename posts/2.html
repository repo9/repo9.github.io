클라이언트-인증자 프로토콜(CTAP)

2. 적합성

비규범으로 표시된 섹션뿐만 아니라 이 사양의 모든 저작 지침, 다이어그램, 예 및 참고 사항은 비규범입니다. 이 사양의 다른 모든 것은 규범적입니다. 키워드 "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY" 및 "OPTIONAL "는 [RFC2119] 에 설명된 대로 해석되어야 합니다 .

3. 프로토콜 구조

이 프로토콜은 세 부분으로 지정됩니다. 인증자 API : 이 추상화 수준에서 각 인증자 작업은 API 호출과 유사하게 정의됩니다. 즉, 입력 매개변수를 수락하고 출력 또는 오류 코드를 반환합니다. 이 API 수준은 개념적이며 실제 API를 나타내지 않습니다. 실제 API는 각 구현 플랫폼에서 제공됩니다. 메시지 인코딩 : 인증자 API에서 메서드를 호출하려면 호스트가 요청을 구성 및 인코딩하고 선택한 전송 프로토콜을 통해 인증자에게 보내야 합니다. 그러면 인증자가 요청을 처리하고 인코딩된 응답을 반환합니다. 전송 관련 바인딩 : 요청 및 응답은 특정 전송(예: USB, NFC, Bluetooth)을 통해 로밍 인증자에게 전달됩니다. 각 전송 기술에 대해 이 프로토콜에 대한 메시지 바인딩이 지정됩니다. 이 문서는 로밍 FIDO2 인증자를 위해 위의 세 부분을 모두 지정합니다.

4. 프로토콜 구조

플랫폼과 인증자 간의 일반적인 프로토콜은 다음과 같습니다. 플랫폼은 인증자와 연결을 설정합니다. 플랫폼은 인증자의 기능을 결정하는 데 도움이 되는 인증자GetInfo 명령을 사용하여 인증자에 대한 정보를 얻습니다. 플랫폼은 인증자가 작업을 지원할 수 있는 경우 작업에 대한 명령을 보냅니다. 인증자는 응답 데이터 또는 오류로 응답합니다.

5. 프로토콜 구조

인증자 API의 각 작업은 다른 작업과 독립적으로 수행될 수 있으며 모든 작업은 비동기식입니다. 인증자는 리소스 사용을 제한하기 위해 미해결 작업에 제한을 적용할 수 있습니다. 이 경우 인증자는 사용 중 상태를 반환하고 호스트는 나중에 작업을 다시 시도해야 합니다. 또한 이 프로토콜은 요청 및 응답을 순서대로 또는 안정적으로 전달하지 않습니다. 이러한 속성이 필요한 경우 기본 전송 프로토콜에서 제공하거나 응용 프로그램에서 상위 계층에서 구현해야 합니다. 이 API 수준은 개념적이며 실제 API를 나타내지 않습니다. 실제 API는 각 구현 플랫폼에서 제공됩니다. 인증자 API에는 다음과 같은 메서드와 데이터 구조가 있습니다.
